## 5. ブラウザ定常フレーム基準値

### 5.1 条件

- `npm run build`
- `npm run preview -- --host 127.0.0.1 --port 4173`
- URL: `/?seed=7&slots=24&q=0&m=1&dpr=1`
- production build
- Chrome 150 / WebGL2 / ANGLE Metal
- 毎試行でhard reload
- reload後15秒warm-up
- 30秒計測
- 5試行
- `?m=1`によりtier 0固定、parallax無効

`Frame`はVSyncを含むrAF間隔、`Update`は`sim.step() × n + renderer.render()`のJavaScript時間である。`Update`はGPU完了時間ではない。

## 6. WebGL instrumented基準値

通常のフレーム試行とは分離し、WebGL draw / buffer uploadを一時wrapperで数え、`EXT_disjoint_timer_query_webgl2`でフレーム全体のGPU command時間を測った。

- 計測時間: 10秒
- フレーム数: 600
- disjoint試行: なし
- 通常のUpdate値とは合算・比較しない

|指標|min|p50|p95|p99|max|mean|
|---|---:|---:|---:|---:|---:|---:|
|draw calls/frame|28|28|28|28|28|28|
|instanced draws/frame|11|11|11|11|11|11|
|submitted vertices/frame|487,794|491,628|503,058|503,076|503,082|493,565|
|`bufferSubData` bytes/frame|27,840|28,480|28,960|29,104|29,152|28,487|
|GPU time/frame|5.922 ms|10.983 ms|14.100 ms|15.005 ms|22.639 ms|10.859 ms|

定常中の`bufferData`は0 bytes/frameだった。GPU p95は16.67 ms枠まで約2.57 msの余裕しかなく、CPUの`Update`よりGPU側の余裕が小さい。draw call削減、Backdrop、Spray、post-processingを優先候補に残す根拠になる。

## 7. baselineから確定した優先順位

1. CPUは`GridDetector.findPairs`が最初の対象。instrumented sim全体の55.57%を占める。
2. GPUはp50 10.98 ms / p95 14.10 msで、60 Hz枠に対する余裕が小さい。
3. draw callは全フレームで28、instanced drawは11。決定論的なafter指標として使える。
4. 動的buffer uploadはp50 28,480 bytes/frame。高リフレッシュレート時のstepなしフレーム抑制を直接判定できる。
5. FPSは60 Hz VSyncで飽和しているため、単独の採否指標にしない。

この時点では改善実装を行っていない。以後は1作業単位ごとにbeforeを再確認し、afterがノイズを超えて改善しなければ採用しない。
