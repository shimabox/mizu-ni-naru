# 初期正式版10分連続動作

## 目的

4時間帯動画、現在時刻による選択、ネイティブ日時表示を組み込んだ正式版が、実機のフルスクリーンプレビューで安定して動作するかをPhase 0と同じ条件で測る。

## 条件

|項目|値|
|:---|:---|
|ブランチ|`feat/macos-video-screensaver`|
|計測日時|2026-07-16 11:31:40〜11:41:40 JST|
|macOS|26.2（Build 25C56）|
|Mac|MacBookAir10,1 / arm64|
|対象|`MizuNiNaru.saver` フルスクリーンプレビュー|
|選択動画|昼 `day.mp4`|
|動画|H.264 High / yuv420p / 1280×720 / 30fps|
|計測時間|600秒|
|間隔|5秒|
|再生プロセスのサンプル数|119|

## 結果

|指標|最小|平均|最大|Phase 0平均との差|
|:---|---:|---:|---:|---:|
|CPU|0.0%|7.13%|15.8%|-1.63pt（-18.6%）|
|RSS|41.38MiB|64.33MiB|66.63MiB|+6.88MiB|

CPU最小値とRSS最小値には、600秒目でユーザーがプレビューを閉じたあとの最終サンプルが含まれる。表示中に目視したRSSは65〜66MiBで横ばいだった。終了による解放を含めた先頭から末尾のRSS変化は-24.83MiBだった。

小型プレビュー用の別プロセスは110サンプルすべてCPU 0.0%で、正式版の再生評価から除外した。

## 終了後

プレビューとシステム設定の終了後もmacOSの待機ホストは残ったが、CPUは直後に0.0%まで低下し、RSSは37.28MiBまで解放された。映像デコードと時計更新が終了後に継続する問題は再発しなかった。

## 比較と判断

10分間の再生中CPU平均はPhase 0の8.76%から7.13%へ1.63ポイント低下した。4時間帯動画とネイティブ日時表示を追加してもCPU悪化はない。

RSS平均はPhase 0より6.88MiB増えた。ただしこの計測時点では旧Phase 0バンドルもインストールされており、同一ホストが両バンドルをロードしていた。正式版の表示中RSSは安定しており、継続的な増加はない。旧バンドル削除後に短時間のクリーン計測を追加し、正式版単体のRSSを確認する。

CPU、10分連続動作、終了時停止は合格。ユーザーの実機確認で、昼の映像、日時、時計、ループの継ぎ目にも問題がないことを確認した。その後、反復感を抑えるため動画を60秒へ延長した。最終素材の結果は[動画の長尺化と画質比較](20260716-video-loop-quality.md)を参照する。

機械可読な集計は [`raw/20260716-formal-10min-summary.json`](raw/20260716-formal-10min-summary.json) に保存した。今後の再計測はリポジトリに追加した `npm run screensaver:benchmark:mac` を使う。

旧Phase 0削除後の最終60秒素材による単体計測は[最終版単体計測](20260716-final-clean-runtime.md)に記録している。
